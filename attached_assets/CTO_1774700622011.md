# Kataleya v2.2 — Technical Architecture Report
### Audience: Chief Technology Officer / DevOps Lead
### Date: March 2026 | Classification: Internal Technical

---

## Executive Summary

Kataleya is a privacy-first React Native mobile application built on Expo SDK 52 with the New Architecture enabled. The codebase comprises **6,500+ lines of source code across 36+ files**, spanning a mobile client, a minimal relay API, and a shared type layer. Every architectural decision is designed to make the privacy guarantee technically verifiable — not just a policy statement.

The v2.2 sprint closed the remaining architecture gaps: the crypto backend was upgraded from pure-JS to native OpenSSL/CommonCrypto, the insights engine now surfaces data-age and sufficiency disclosures, the onboarding flow is complete (7 steps including privacy explanation and notification permission), push notifications are wired end-to-end, and the sponsor check-in has an offline queue with automatic replay. The restlessness field is now live-wired to accelerometer RMS — it is no longer a static zero.

## 3. Three-Vault Storage Architecture

### Tier 1 — Surface (AsyncStorage)
**Keys (canonical, namespaced — complete list):**
```
kataleya.surface.sobriety_start
kataleya.surface.onboarded
kataleya.surface.name
kataleya.surface.substance
kataleya.surface.burn_log              # tombstone: written before burnAll(), cleared after
kataleya.surface.pending_checkin       # offline check-in queue, replayed on foreground
kataleya.surface.notif_enabled         # boolean ('1'/'0')
kataleya.surface.notif_reminder_time   # 'HH:MM', default '20:00'
kataleya.surface.last_notified_milestone # integer — prevents re-firing milestone alerts
```
**Data held:** Sobriety start date, display name, tracked substance, onboarding state, burn tombstone, offline check-in queue, notification preferences.
**Encryption:** OS-level file encryption (device passcode). Not hardware-backed.

### Tier 2 — Sanctuary (SQLite)
**Engine:** `expo-sqlite` v56, WAL mode, foreign key enforcement
**Schema:** Three tables — `mood_logs`, `journal_entries`, `circadian_log`
**Key constraints:**
```sql
mood_score   INTEGER CHECK (mood_score BETWEEN 1 AND 10)
restlessness REAL    CHECK (restlessness >= 0 AND restlessness <= 1)
correction   TEXT    -- user override field; schema-level, not app-level
```
**Indexes:** `ts DESC` on all three tables.
**Migration system:** `PRAGMA user_version` guard introduced. Current schema is `SCHEMA_VERSION = 1`. Migrations run inside a `migrate()` function called at database init. Safe for future schema changes without data loss.
**Data held:** All health-adjacent data. Never leaves the device. Never referenced by any server endpoint.

### Tier 3 — Fortress (SecureStore)
**Engine:** `expo-secure-store` v56
**iOS backing:** iOS Keychain Services
**Android backing:** Android Keystore System (AES-256, hardware-backed where supported)
**Keys:**
```
kataleya.fortress.sponsor_channel_id
kataleya.fortress.sponsor_token
kataleya.fortress.sponsor_role
kataleya.fortress.sponsor_last_poll
kataleya.fortress.sponsor_invite
```
**Data held:** Sponsor channel credentials only.

### Vault Separation Enforcement

`utils/storage.ts` exports `Surface`, `Sanctuary`, and `Fortress` as the only interfaces. Direct AsyncStorage/SecureStore/SQLite calls are not available to the rest of the codebase. `SponsorVault` and `PrivateVault` are `@deprecated` shims delegating to the canonical vault objects — ensuring migration completeness without runtime breakage.

## 4. Zero-Knowledge Encrypted Backup

**File:** `utils/backup.ts` (306 lines)
**Crypto backend:** `react-native-quick-crypto` (JSI native module — OpenSSL on Android, CommonCrypto on iOS)

### Upgrade from v2.1

The previous implementation used `crypto-js` (pure JavaScript PBKDF2). It was replaced with `react-native-quick-crypto` for three reasons:

1. **PBKDF2 key derivation runs in native code.** 100,000 iterations no longer expand through the JS heap. Key state is computed in C++ (OpenSSL/CommonCrypto) and surfaces as a single Buffer.
2. **OS CSPRNG.** `randomBytes()` routes to `/dev/urandom` (Linux/Android) and `SecureRandom` (iOS), not a userspace PRNG.
3. **Constant-time MAC comparison.** `timingSafeEqual()` replaces string comparison, eliminating the timing side-channel on HMAC verification.

The encrypted blob format (`{v, alg, iters, salt, iv, mac, data}`) is unchanged. Backups created under v2.1 restore correctly under v2.2.

### Design

```
User passphrase
    ↓  PBKDF2-SHA256 (100,000 iterations, 32-byte random salt) — native OpenSSL
AES-256-CBC key  →  encrypt payload JSON (native AES)
    ↓  HMAC-SHA256 over ciphertext (Encrypt-then-MAC, timingSafeEqual verification)
Encrypted blob: { v, alg, iters, salt, iv, mac, data }
    ↓  saveAndShareBackup()
Native share sheet → user saves to iCloud Drive / Files / Google Drive
```

### Zero-knowledge properties

- The passphrase is used to derive a key, then exits scope. It is never stored, never logged, never sent anywhere.
- The encrypted blob is opaque: `{ v, alg, iters, salt, iv, mac, data }` — no plaintext fields.
- Cloud providers (iCloud, Google Drive) store the encrypted blob only. Without the passphrase, the content is unrecoverable — including by us.
- The HMAC check happens before decryption, using `timingSafeEqual()` (constant-time). A wrong passphrase produces a wrong key → wrong HMAC → error thrown before `createDecipheriv` is called.

### Platform requirement

`react-native-quick-crypto` is a JSI native module. Calling `encryptBackup()` or `decryptBackup()` inside Expo Go throws a clear error at runtime. Backup functionality requires an EAS Build or local development build. This constraint is documented in the vault UI ("Requires a native device build").

### What is backed up

- Sanctuary: `mood_logs`, `journal_entries`, `circadian_log` (up to 5,000 / 2,000 / 365 entries)
- Surface: `sobriety_start`, `name`, `substance`

### What is NOT backed up

- Fortress (sponsor credentials): channels are ephemeral. They expire before any restore is useful.
- Notification preferences and offline queue: device-local configuration, not health data.

### Performance note

PBKDF2 at 100,000 iterations in native code takes approximately 1–2 seconds on a mid-range device (compared to 2–4 seconds in the pure-JS implementation). A loading spinner is shown during both encrypt and decrypt operations. The iteration count is what makes brute-force attacks impractical.

## 5. Circadian Engine

**File:** `hooks/useCircadian.ts` | 68 lines
**Phases:** `dawn` (05:00–08:00) | `day` (08:00–17:00) | `goldenHour` (17:00–20:00) | `night` (20:00–05:00)
**Adaptive poll:** 10 seconds during transitions, next-minute boundary during stable phases
**Theme interpolation:** 0.0 (MorningBloom) → 1.0 (MidnightGarden), computed locally on every tick
**No network call.** Phase is computed from `getCurrentMinutes()` — a local clock read.

## 6. Insights Engine

**File:** `hooks/useInsights.ts` | 219 lines

**Confidence formula:**
```
sizeScore  = 1 - 1 / (1 + n / 12)        // sigmoid
recency    = exp(-ageDays / 45)           // half-life 45 days
confidence = min(0.97, sizeScore × recency)
```

**Surfacing threshold:** 0.72 — a named export, independently testable.
**Pattern types:** `phaseLowestMood`, `moodTrend` (14-day), `checkInFrequency`
**Computation:** Fully local. No network call. Runs once on mount. Fails silently (insights are enhancement, not core).

**v2.2 additions:**
- `hasEnoughData`: true when ≥7 mood logs exist in the last 90-day window. The Growth tab shows an empty-state prompt rather than an empty list when this is false.
- `dataAgeDays`: integer age of the most recent log. When > 30, a nudge appears: "Based on your logs from X days ago. Log today to refresh."

**Restlessness field:** The `restlessness` column in `mood_logs` is now populated from live accelerometer data. `useOrchidSway` computes a 100-sample rolling RMS over a 5-second window (WINDOW_SIZE=100, MAX_DELTA=0.204g) and exports `restlessnessScore` (0.0–1.0). This value is written to `mood_logs` at save time. The `restlessness: 0` placeholder has been removed.

## 7. Sponsor Relay Architecture

**Server:** Express 5 + Pino, hosted on Replit infrastructure
**State:** In-memory only. No database. No log of signals.
**Protocol:** Short-poll, 5-second interval
**Channel lifecycle:** 48h unconnected, 90d from last activity once connected

**Server stores per channel (complete list):**
```typescript
{
  id: string,              // random UUID
  inviteCode: string,      // 6-char alphanumeric
  userToken: string,       // 48 hex bytes
  sponsorToken: string,    // 48 hex bytes
  connected: boolean,
  checkedIn: boolean,
  checkedInDate: string | null,   // e.g. "Mon Mar 28 2026"
  orchidStage: string,            // e.g. "seedling"
  milestones: number,             // integer
  presenceQueue: PresenceSignal[], // max 50 signals
  createdAt: number,
  expiresAt: number,
}
```

**What the server does NOT store:** User name, sobriety date, mood score, journal content, circadian history, device identifier, location.

### Offline check-in queue (v2.2)

If the device is offline at check-in time, `sendCheckIn()` catches the network failure and writes the pending check-in to `Surface.PENDING_CHECKIN` (AsyncStorage) rather than failing silently. On every app foreground event (`AppState 'active'`), the hook checks for a queued check-in and replays it. Stale entries (> 24 hours) are discarded. The `hasPendingCheckin` flag is exposed from the hook for UI display.

### Rate limiting (server-side)

Per-token rate limits enforced via in-memory Maps:
- Poll: 30 requests / 60 seconds
- Presence signal: 10 requests / 60 seconds
- Check-in: 5 requests / 60 seconds

## 8. Push Notifications

**Hook:** `hooks/useNotifications.ts` | 149 lines
**Package:** `expo-notifications` (declared in `app.json` plugins)
**Permission:** Requested during onboarding step 5 ("notifications"). User may decline; preferences stored in `Surface.NOTIF_ENABLED`.

**Two notification types:**

| Type | Trigger | Content |
|------|---------|---------|
| Daily reminder | Scheduled daily at user-configured time (default 20:00) | "How are you feeling? A moment of honesty is waiting in your sanctuary." |
| Milestone bloom | `daysSober` hits a milestone day (1/7/14/30/60/90/180/365/500/730/1000) | Day-specific message, e.g. "Thirty days. The orchid has bloomed." |

Milestone notifications fire at most once per milestone. `Surface.LAST_NOTIFIED_MILESTONE` prevents re-triggering across app restarts.
