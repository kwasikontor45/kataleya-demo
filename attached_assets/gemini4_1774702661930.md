**2\. Block A: Offline Check-in Queue (Implementation)**

**Files:** hooks/useSponsorChannel.ts, utils/storage.ts

Now moving to the "Resiliency" block for the Sponsor Channel. We are implementing a **single-slot pending queue** to handle intermittent connectivity without data loss.

**Logic implemented:**

* **Storage:** We use kataleya.surface.pending\_checkin as our volatile queue.  
* **TTL:** 24-hour expiration check.  
* **Trigger:** Replays on app\_foreground or when the poll reconnects.

// hooks/useSponsorChannel.ts

const PENDING\_KEY \= 'kataleya.surface.pending\_checkin';  
const TTL\_MS \= 24 \* 60 \* 60 \* 1000; // 24 Hours

export const useSponsorChannel \= () \=\> {  
  const \[isSyncing, setIsSyncing\] \= useState(false);

  // 1\. REPLAY LOGIC  
  const replayPendingCheckIn \= async () \=\> {  
    const raw \= await Surface.get(PENDING\_KEY);  
    if (\!raw) return;

    const pending \= JSON.parse(raw);  
    const isExpired \= Date.now() \- pending.ts \> TTL\_MS;

    if (isExpired) {  
      await Surface.delete(PENDING\_KEY);  
      return;  
    }

    try {  
      setIsSyncing(true);  
      await sendCheckInToServer(pending.data); // Internal fetch helper  
      await Surface.delete(PENDING\_KEY); // Success: Clear queue  
    } catch (e) {  
      // Still offline, keep in queue for next attempt  
    } finally {  
      setIsSyncing(false);  
    }  
  };

  // 2\. MODIFIED CHECK-IN  
  const performCheckIn \= async (orchidStage: string, milestones: number) \=\> {  
    const data \= { orchidStage, milestones, ts: Date.now() };

    try {  
      await sendCheckInToServer(data);  
    } catch (error) {  
      // Network failure: Queue it  
      await Surface.set(PENDING\_KEY, JSON.stringify({  
        ts: Date.now(),  
        data  
      }));  
    }  
  };

  // 3\. RECONNECTION TRIGGERS  
  useEffect(() \=\> {  
    // Replay on mount and whenever connection is restored  
    if (isConnected) replayPendingCheckIn();  
  }, \[isConnected\]);

  return { performCheckIn, isSyncing, hasPending: \!\!Surface.get(PENDING\_KEY) };  
};

**CTO Confirmation: The 24-Hour Threshold**

**Q: What happens if the user is offline for more than 24 hours before reconnecting?**

**A: The check-in is discarded.** In a recovery context, a stale "I'm okay" signal from two days ago is dangerous misinformation for a sponsor. If the connection isn't restored within the **24-hour clinical window**, we purge the signal silently to ensure that when the user *does* finally reconnect, they send a fresh, accurate reflection of their current state.
