## **What's genuinely impressive**

### **The privacy architecture is real and well-thought-out. The three-vault separation with enforced module boundaries, the zero-knowledge backup with Encrypt-then-MAC, HMAC check before decryption, the fact that the SQLite schema has no `user_id` column — these aren't just talking points, they're structurally correct decisions. The sponsor channel audit trail (what *is* and *isn't* sent) is unusually honest for a pitch document.**

### ---

## **What I'd change for production**

### **🔴 Critical**

### **1\. `crypto-js` is the wrong library for this threat model. It's pure JavaScript, meaning no access to the device's hardware security module. For a sobriety app where you're claiming AES-256 as a trust anchor, you should be using `expo-crypto` or React Native's bridge to the OS crypto APIs (which use the Secure Enclave on iOS, StrongBox on Android). `crypto-js` also has a known history of weak RNG on some environments. The 2–4 second PBKDF2 delay they document is partly a symptom of this — native crypto does it in milliseconds.**

### **2\. No SQLite migration system is a ticking bomb. They acknowledge it themselves as "medium severity" debt, but it's worse than that. Any schema change without `user_version` PRAGMA guards risks silent data corruption or crashes on upgrade for existing users. This needs to land *before* any production release, not v2.1.**

### **3\. The web fallback is a privacy hole. `AsyncStorage` as a stub for SQLite and SecureStore on web is documented as "dev/preview only," but there's no enforcement at the type level or runtime that prevents a web build from reaching production. This needs a hard `throw` or build-time guard, not a comment.**

### **4\. `restorePayload` does sequential awaits in a loop.**

### **typescript**

### **for (const log of \[...payload.sanctuary.moodLogs\].reverse()) {**

###   **await Sanctuary.saveMoodLog(...)**

### **}**

### **For a user with 5,000 mood logs this is going to take a very long time and will likely be killed by the OS if the app backgrounds during restore. Needs batched SQLite transactions (`BEGIN TRANSACTION / COMMIT`).**

### ---

### **🟡 Important**

### **5\. The `migrate()` function will silently no-op on existing installs. `CREATE TABLE IF NOT EXISTS` only creates tables — it doesn't add columns to existing ones. So if you add a column in v2.1 (say, a `tags` field on `journal_entries`), users upgrading from v2.0 won't get it and won't get an error — their queries will just fail at runtime. You need the `user_version` PRAGMA system *with* `ALTER TABLE` migrations.**

### **6\. The Burn Ritual has no confirmation outside of the 3-second hold. A 3-second hold in a moment of distress (which is exactly when someone in recovery might reach for it) is a low bar. I'd add a modal confirmation step, or at minimum log the event locally before the wipe so there's a grace window. The irreversibility is by design, but the UX needs to respect that the user might be shaking.**

### **7\. PBKDF2 iteration count is hardcoded. 100,000 iterations was appropriate in \~2021. OWASP's current recommendation is 600,000 for SHA-256. More importantly, the iteration count is stored in the blob (`iters: 100_000`), which means you *can* upgrade this — but only if you actually read and use `blob.iters` during decryption instead of hardcoding it. The current `deriveKey()` call should use `blob.iters`, not the constant.**

### **8\. No rate limiting or abuse protection on the sponsor relay. The in-memory relay has no authentication beyond the token, no rate limiting on the poll endpoint, and the presence queue is capped at 50 signals but with no time-based expiry. Someone with a valid token could hammer the poll endpoint. For a Replit-hosted server this is a real risk.**

### ---

### **🟢 Nice to have**

### **9\. The insights confidence formula uses a 45-day half-life that's undocumented to the user. Users who go 3 weeks without logging will see their insights disappear with no explanation. Surface this in the UI.**

### **10\. `restlessness` is always 0 — they note it themselves. The accelerometer data is being captured (`useOrchidSway`) but never mapped to the field. This is easy to wire up and would make the insights engine meaningfully richer.**

### **11\. No offline-first conflict resolution for the sponsor channel. If the user checks in while offline, does that check-in queue? Currently it appears to just fail silently.**

### ---

## **The bottom line**

### **The architecture is legitimate and the privacy claims are defensible. The gap between "working prototype" and "production ready" here is mostly about upgrade safety (migrations), crypto correctness (native APIs instead of pure JS), and recovery UX (the burn ritual and restore flow both need more care). None of these are fundamental rethinks — they're the unglamorous hardening work that separates a compelling demo from something you'd trust with years of someone's recovery history.**

However, "Production Ready" in a recovery context is a high bar. To move from **85% completion** to a stable public launch, follow this strict **Production-Ready Guide**.

---

## **1\. Eliminate High-Priority Technical Debt**

Before the first public build, we must address the "Medium" severity items in the debt log:

* **Implement SQLite Migration Logic:** We currently have no `user_version` PRAGMA guard. If we change the schema in v2.2, users will lose their entire "Sanctuary" history upon update. This is unacceptable for a paid "Bloom" tier.  
* **Pin SQLite Versions:** Ensure we are pinned to the stable Expo SDK 52 GA to prevent runtime crashes on newer Android/iOS versions.

## **2\. Harden the Privacy Claims (Audit-Proofing)**

The Board and Investors are relying on the "Physically Cannot Access" claim.

* **The "Sponsor Relay" Disclosure:** Ensure the UI explicitly labels the "Check-in date" and "Orchid stage" as opt-in shared data. The code confirms these are held only in RAM and are not health metrics, but the user must *know* this during the invite process.  
* **E2E for Sponsor Messages:** The relay currently relies on TLS for presence signals (Water/Light). Before we launch the **Sponsor Message Channel (v3.0)**, we must implement E2E encryption so the server never sees actual message text.

## **3\. Performance & UX Optimization**

* **The "PBKDF2" Spinner:** Since key derivation takes 2–4 seconds on mid-range devices, the "Vault" UI must not hang. Ensure the loading spinner is non-blocking for other background tasks.  
* **Circadian Adaptive Polling:** Verify that the 10-second polling during "Golden Hour" transitions doesn't drain the battery. We need to confirm the JS thread wakeups are as efficient as the documentation claims.

## **4\. Business & Regulatory Readiness**

* **Subscription Layer:** We have zero progress on the payment layer. We need to integrate **RevenueCat** or a similar service that mirrors our "Privacy First" approach—ensuring we track subscriptions without tying them to health logs.  
* **The "Burn Ritual" Validation:** This is our "Panic Button." We must perform a destructive test on all native platforms (iOS/Android) to ensure `SecureStore.deleteItemAsync` and the SQLite `DELETE` commands leave no forensic trace in the app's cache.
